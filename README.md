# Omni-Disciplinary Knowledge Graph

A self-organizing personal knowledge base powered by AI agents in VS Code or the terminal. Inspired by Andrej Karpathy's "LLM Wiki" concept.

No vector databases. No graph databases. No servers. Just Markdown files, a rules engine, and an AI that builds and queries the graph for you.

## The Idea

You accumulate knowledge across a dozen domains — papers you read, meeting notes, code you studied, datasets you explored, ideas you had. It lives in scattered files and fades from memory.

This system turns all of it into a connected, queryable knowledge graph that an AI agent maintains and reasons over. Drop in a file. Tell the agent to process it. Come back later and ask a question — it retrieves the relevant context from your own graph before answering, not from its training data.

It works for anything: a research project, a hobby deep-dive, a startup, a novel you're writing, a personal reading log, competitive analysis, a D&D campaign. Any domain, any mix of domains.

## How It Works End to End

```
Clone repo
    ↓
Tell your agent: "Read bootstrap.md and execute the instructions."
    ↓
Agent builds your workspace (folders, indexes, rules engine)
    ↓
Drop files into 01_raw_inputs/ — PDFs, notes, CSVs, code, transcripts, anything
    ↓
Tell your agent: "Process new data"
    ↓
Agent extracts concepts, writes structured nodes, links related ideas, flags contradictions
    ↓
Close your editor. Come back tomorrow. Start a new session.
    ↓
Ask anything: "Query the graph: [your question]"
    ↓
Agent retrieves from your graph — fills gaps from model knowledge if needed
    ↓
You get a grounded answer from your own accumulated knowledge
    ↓
Keep adding files. Keep asking. Keep building.
```

## Commands

### `Process new data`
Scans `01_raw_inputs/` for new files, extracts structured knowledge into nodes, links related concepts, and updates all indexes. Prompts for clearance level when the content type is ambiguous.

### `Query the graph: [question]`
Standard query — up to 8 nodes. Retrieves from the local graph first. If coverage is insufficient, fills the gap using the model's trained knowledge (labelled clearly). Returns confidence label, sources, and any gaps.

```
Query the graph: What are the common failure modes across the architecture papers I uploaded?
Query the graph: What's the strongest counterargument to my current thesis?
Query the graph: What did I learn about fermentation that applies to this new recipe idea?
```

### `Query the graph [deep]: [question]`
Same but reads up to 15 nodes. Use for complex or cross-domain questions.

### `Synthesize across domains`
Finds nodes from different disciplines and writes a cross-domain report. The synthesis inherits the highest clearance level of any source node.

### `Sync graph`
Run at the start of any session where `01_raw_inputs/` may have changed:
- **New** → processes and creates a node
- **Updated** → re-extracts, rewrites, re-confirms clearance if content changed significantly
- **Deleted** → removes the node, cascades cleanup to all references
- **Unchanged** → skipped
- **Broken links** → reports dead `[[WikiLinks]]` without auto-deleting

### `Resolve contradiction: [node A] vs [node B]`
Reads both nodes, identifies the conflicting claims, searches for evidence, writes a resolution. Classifies as outdated, contextual, or genuinely open. Synthesis inherits the highest clearance of the two nodes.

### `Search external: [topic]`
Forces a web search (only works if `gap_fill_mode: external` is set in `source_config.md`). Applies the full sanitization protocol before any query leaves the local environment.

### `Compress node: [node name]`
Rewrites a verbose node body as a compact bullet-point list. YAML stays untouched.

### `Set clearance: [node name] to [level]`
Updates the clearance field on a node and its registry row. Also flags any synthesis reports that reference the node for review.

## Data Privacy & Security

### Clearance Levels

Every node carries a `clearance` field:

| Level | Meaning | External search |
|---|---|---|
| `public` | No restrictions | Safe to use directly in queries |
| `internal` | Stays within the graph | Must be abstracted before external use |
| `confidential` | Sensitive content | **Never** used to inform external queries |
| `external` | Sourced from outside | Always `confidence: low` |

Synthesis reports automatically inherit the highest clearance of any source node.

### Gap-Fill Modes

Configured in `03_indexes/source_config.md`:

| Mode | Behaviour | Data risk |
|---|---|---|
| `parametric` *(default)* | LLM answers from its trained knowledge | Zero — nothing leaves local environment |
| `external` | Searches Wikipedia, ArXiv, DuckDuckGo | Low — queries are sanitized before sending |
| `none` | Reports the gap only | Zero |

### Query Sanitization (when `external` mode is enabled)

Before any query leaves the local environment the agent must:

1. **Clearance check** — if any consulted node is `confidential`, external search is blocked entirely
2. **Abstraction** — internal project names, employee names, proprietary codenames, and specific financial figures are forbidden in external queries. They must be abstracted to their generic underlying concept:
   - `"Project Orion SSO failure"` → `"Single Sign-On implementation failure analysis"`
   - `"Q3 Helios revenue shortfall"` → `"revenue forecasting gap analysis techniques"`
3. **Fallback** — if a term cannot be safely abstracted, the agent falls back to `parametric` mode automatically

### GDPR-Inspired Principles

The system is designed around five principles drawn from GDPR:

1. **Data minimisation** — extract concepts, not verbatim source text
2. **Purpose limitation** — clearance level governs what a node can be used for
3. **Accuracy** — `last_verified` tracks staleness; nodes older than 6 months are flagged
4. **Right to erasure** — deletion cascades to all references across the graph
5. **No unnecessary external transmission** — `parametric` is the default; `external` requires explicit opt-in

## How Retrieval Works

Retrieval logic lives in `03_indexes/retrieval_protocol.md`, loaded only when a query runs.

**Step 0 — HyDE**: Generates a hypothetical summary of what the ideal answer node would say. Used as a second match signal alongside raw keywords.

**Step 1 — Cluster Pre-filter**: Reads `cluster_index.md`. Narrows to 1–3 disciplines before scanning the full registry.

**Step 2 — Registry Scan**: Scores each row against both raw query and HyDE summary. Assesses confidence: sufficient / partial / insufficient.

**Step 3 — Tiered Node Read**: YAML block only (lines 1–30) first. Full read only if confirmed relevant. Checks clearance. Flags stale nodes. Budget: 8 normal / 15 deep.

**Step 4 — Connection Traversal**: Follows `connections:` and `contradicts:` up to 2 hops when relevant.

**Step 5 — Gap Fill**: Parametric by default (model knowledge, zero leakage). External search requires opt-in and runs sanitization before any query leaves the machine.

**Step 6 — Answer + Log**: Returns confidence · answer · sources · model-knowledge flag · gaps. Logs to `query_log.md`.

## Node Structure

```yaml
title: ""
type: ""            # research_paper | strategy | codebase | transcript | abstract_concept | dataset | synthesis | external
discipline: ""
clearance: ""       # public | internal | confidential | external
tags: []
keywords: []        # specific terms, entities, acronyms — secondary retrieval signal
summary: ""         # one sentence, mandatory
assumptions: []
connections: []     # [[WikiLink]] to related nodes
contradicts: []     # [[WikiLink]] to conflicting nodes
source: ""
confidence: ""      # high | medium | low
date_added: ""      # set once, never changed
last_verified: ""   # updated whenever the node is confirmed current
```
